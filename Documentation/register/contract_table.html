API Contract Table — User Registration

Course: 960121 Web Technology  |  Session: 06 — Security & Auth Logic  |  Feature: POST /api/register  |  Last Updated: 2026-05-05

Endpoint Overview

ComponentDetails
MethodPOST
Endpoint/api/register
DescriptionRegisters a new user account. Validates all fields, checks for duplicate email, hashes the password with bcrypt, and saves the user to auth_user.json.
Auth RequiredNone
Content-Typeapplication/json

Request Body

FieldTypeRequiredValidation Rules
nameString Yes Must be a non-empty string
emailString Yes Valid email format (contains @ and .); must be unique in the system
passwordString Yes Minimum 8 characters; at least 1 uppercase letter (A–Z); at least 1 special character from !@#$%^&*

Example Request Body:

POST /api/register HTTP/1.1
Content-Type: application/json

{
  "name": "Alice Smith",
  "email": "alice@example.com",
  "password": "Secret@1"
}

Response Codes

CodeMeaningResponse Body
201 Created Registration successful 
{ "message": "User registered successfully",
  "user": { "id": "...", "name": "Alice Smith",
            "email": "alice@example.com",
            "registrationDate": "2026-05-05T..." } }
400 Bad Request Missing field or password does not meet rules 
{ "error": "Password must be at least 8 characters,
           include 1 uppercase letter and 1 special character" }
409 Conflict Email already registered 
{ "error": "Email already registered" }
500 Internal Server Error Unexpected server error (e.g., file I/O failure) 
{ "error": "Internal server error" }

Security Notes

ConcernMitigation
Password storagePasswords are never stored in plain text. Hashed with bcrypt (salt rounds: 10) before saving.
Transport securityAll API calls must be made over HTTPS in production to prevent interception of credentials in transit.
Duplicate preventionEmail uniqueness is checked before inserting a new record; returns 409 Conflict if already exists.
Input validationAll fields are validated on both frontend and backend before any database operation is performed.
Response leakagePassword hash is never returned in the response. Only id, name, email, and registrationDate are sent back.

Data Saved to auth_user.json

{
  "id": "<uuid-v4>",
  "name": "Alice Smith",
  "email": "alice@example.com",
  "password": "$2b$10$...<bcrypt-hash>...",
  "registrationDate": "2026-05-05T10:30:00.000Z"
}
The password field is always a bcrypt hash — the plain-text password is never saved.
